Chapter 1 : Introduction

1.1 Description of the current situation and opportunity:
in jordan, local municipalities and the GAM are responsible for managing a massive amount of public infrastructure. this includes keeping roads paved, fixing streetlights, maintaining public sanitation, and managing safety hazards.
Right now ,when something breaks, like a broken streetlight, there is no single way for citizens to report it. people usually try calling a municipal hotline,posting of facebook groups, or visting a local office to complain
on the inside,municipal offices still manage their maintenance manual processes,paper work orders,or basic phone calls.
this creates a great opportunity for a digital solution, by creating a unified platfrom that connects citizen reports directly to an automated backend scheduling and tracking system.

1.2 Problem Statement
the way civic maintenance is handled right now has three main problems:
1. no clear way for citizens to report issues :  citizens do not have a simple tool to report public damage
2. bad resource management : dispatchers cannot easily see which maintenance teams are busy,or where are they located. (poor scheduling, wasted time,blind assignments)
3. no tracking : there is no automated system to track how long a ticket has been ignored

1.3 proposed solution and system overview:
to solve these problems, we proposed BaladiyaConnect, a web-based service routing and ticket management system
acts as an aoutomated bridge between the public and the municipal crews. when a citizen reports an issue , the system automatically reads the location,type of issue, then checks the database to see which maintenance crew is closest and has the right speciality.
finally, it assigns the task directly to that team's queue,lets the citizen track the progress. this eliminates manual phone calls,paper work, turning a slow process into an automated digtial workflow

1.4 project overview
the main goals of the BaladiyaConnect system are :
1. to create a reliable central database that safely stores and organizes all citizen reports
2. to build an intelligent routing feature that automatically assigns tasks to maintenece crews based on their current location,speciality
3. to design a live tracking and notification subsystem that lets citizen query theri ticket progress
4. to build an automated escalation feature that flags urgent , unfinished tasks and alerts senior engineers if a deadline is missed
5. to design an adminstrative dashboard that shows municipal directors clear charts on average fix times,common problem areas , overall team performance

Chapter 2: Requirements and Analysis

2.1 Software Process Model used : the Iterative Development Model will be used
instead of trying to build the entire system all at once, the Iterative model allows us to design, implement, test the project in smaller cycles.
in the first iteration, the focus will be on setting up the core database structure, foundational features such as submitting a report and allowing field crews to update a ticket's status
once these basic operations are stable,subsequent iterations will introduce the more complex backend logic, including the automated crew-assignment algorithm,the live citizen tracking system,and the automated SLA escalation background processes

2.2 List of Funcional and Non-Functional Requirements:

2.2.1 FRs:
FR-1 (Issue Reporting): The system shall allow citizens to submit a complaint by selecting a defined category, entering a description, and selecting their neighborhood district
FR-2 (Automatic assignment): the system shall automatically read incoming unassigned tickets , find the closest repair crew that has the right skills,add the job to the team with the shortest to-do list (queue)
FR-3 (Crew Task List) : the system shall allow the field crew supervisor to log into a secure dashboard, to view their daily list of assigned tasks
FR-4 (Citizen Status Tracker): the system shall allow the citizen to input a unique ticket number to view the live progress of their complaint, and the system shall send automated notification updates to the citizen when the ticket status changes
FR-5 (Urgent Deadlines): the system shall monitor the duration of open tickets and shall automatically transmit an alert notification to senior engineers if a high priority safety hazard isnt resolved within its deadline
FR-6 (Manager Charts): the system shall generate performance charts and data reports to allow the municipal director to track average resolution times

2.2.2 Non Functional Requirements
Usability (how easy it is to use)
NFR-1 (Mobile Responsiveness): The system shall automatically adjust its user interface layout to be fully readable and usable on both web browsers and smartphones screens
NFR-2 (System Availabilty): The system shall remain operational and accessible to citizens and muncipal staff 24 hours a day, 7 days a week
NFR-3 (Data Integerity) : The system shall ensure that no data is lost or corrupted if a connection drops while a citizen is submitting a report
